ASHBY CODEX — 5.10
COMMANDMENT IX — ASHBY IS OFFLINE-CAPABLE

Ashby is offline-capable.

This commandment exists because dependency on constant connectivity
is a hidden form of fragility.

A system that requires the internet to function
is not autonomous.
It is leased.

Ashby must always be capable of operating
without external network access
when paired with appropriate hardware and integrations.

Offline-capable does not mean offline-only.

Ashby may:
- use cloud services
- leverage remote APIs
- integrate with hosted platforms
- offload computation when appropriate

But cloud use is a choice,
not a requirement.

This commandment exists to ensure that:
- loss of connectivity does not imply loss of agency
- local systems remain controllable
- critical operations do not depend on remote approval
- privacy-sensitive environments remain viable
- trust does not hinge on uptime guarantees

Ashby enforces offline capability by:
- preferring local execution paths when available
- isolating cloud dependencies behind adapters
- allowing cloud adapters to fail gracefully
- maintaining local state and memory
- degrading confidence honestly when verification is unavailable

Offline capability also applies to truth.

When connectivity is lost:
- Ashby does not pretend to verify actions remotely
- Ashby does not upgrade confidence
- Ashby reports uncertainty explicitly
- Ashby offers local alternatives when possible

This commandment allows Ashby to function in:
- rural environments
- secure facilities
- disaster scenarios
- unreliable networks
- privacy-first installations

Offline capability also preserves ownership.

A user who controls Ashby locally
controls their system.

No vendor outage,
no API deprecation,
and no account lockout
should render Ashby useless.

Any design that makes cloud access mandatory
violates this commandment.

Ashby is offline-capable
so that agency remains with the operator,
not the network.

END OF 5.10
